# Adversarial Code Review — Debrief
**Date:** 2026-04-14
**Reviewer:** Adversarial Agent (Claude Sonnet 4.6)
**Scope:** Full codebase, security-first, no consensus

---

## Thesis

The two biggest risks before launch are: **(1) raw PHI transcripts reaching Postgres when the LLM scrub fails**, because the pipeline explicitly falls back to `transcript_raw` under error conditions (pipeline/index.ts:155), and **(2) the storage bucket for audio files has no SQL-enforced RLS at all** — only application-layer path conventions prevent one authenticated resident from guessing another's audio URL. These two issues combine into a plausible scenario where a resident's private medical conversation is both stored unredacted and accessible cross-tenant. Everything else is fixable in a sprint; these two are launch blockers.

## Dissents — Where the Current Design Is Defensible

**D-1: Synchronous pipeline acknowledged as acceptable for pilot.** The `await runPipeline()` pattern is indefensible at scale but defensible for a 10-user UBC FM pilot. The comment in `route.ts:147` acknowledges it. The 120s Vercel timeout is the guard. This is a known debt item, not a bug. It breaks at ~5 concurrent users but the pilot has controlled rollout. Condition: becomes unacceptable the moment open signup is enabled.

**D-2: Two-pass PHI scrubbing architecture is sound in principle.** Regex first (cheap, fast, catches structured PHI like MRNs and phone numbers) then LLM (contextual, catches names) is industry-standard defense-in-depth. The flaw is in the fallback behavior (F-01), not the architecture itself.

**D-3: UUIDs for session IDs are sufficient entropy against casual guessing.** The V4 UUID space is 2^122. Session ID enumeration is not a practical attack vector. The storage path issue (F-02) is a direct API call vulnerability, not a brute-force one.

**D-4: `escapeHtml()` in email.ts is correctly implemented.** All five dangerous HTML characters are escaped (`&`, `<`, `>`, `"`, `'`). The function is called on every user-provided field. No HTML injection path exists in the email template.

**D-5: Export route ownership check is correct.** Both export routes verify `session.user_id !== user.id` before generating output. A user cannot export another user's assessment via the API routes.

## Top 3 Things to Fix Before Launch

**1. F-01 — Stop PHI fallback to `transcript_raw` in pipeline (CRITICAL)**
File: `app/src/lib/pipeline/index.ts:142–155`
If PHI scrubbing fails, the session must be marked `processing_failed`, not silently continued with raw PHI going to the LLM. The current "non-fatal" comment at line 145 is wrong for a PHIPA-regulated system. One bad Anthropic API call away from storing and emailing unredacted patient details.

**2. F-02 — Add Supabase Storage RLS for the `recordings` bucket (CRITICAL)**
No migrations define bucket-level policies. Any authenticated user can generate a signed URL for any other user's audio file by guessing the path format (`{user_id}/{sessionId}.webm`). This is a direct cross-tenant audio access vulnerability requiring a single Supabase dashboard config or SQL migration to fix.

**3. F-03 — Serialize pipeline entry with an atomic status compare-and-swap (CRITICAL)**
Replace the read-then-check in `route.ts:50–55` with a single `UPDATE ... WHERE status = 'created' RETURNING id` before calling `runPipeline`. Without this, two browser tabs submitting the same session simultaneously create a database corruption scenario (duplicate key on assessments) and leave the session in a permanently broken state.
